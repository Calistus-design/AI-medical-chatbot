# Supabase Database Setup Guide

This guide contains all the necessary SQL scripts to set up the database for the AI First-Aid Assistant project. Run these scripts in your Supabase SQL Editor in the order they appear.

---

# Step 1: Enable PostGIS Extension

Before running any SQL, you must enable the PostGIS extension.

1.  In your Supabase project dashboard, go to the **Database** section.
2.  Click on **Extensions**.
3.  Search for `postgis` and click to **enable** it.

**NB:** if you can't find it run the following sql command in the sql editor

```sql
CREATE EXTENSION IF NOT EXISTS postgis;
```

# Step 2: Create Tables (Conversations & Messages)

This script creates the `conversations` and `messages` tables and sets up their relationship.

*Copy the SQL from your "Conversations & Messages Schema with RLS" query and paste it here.*

```sql
-- File: Supabase SQL Editor

-- Step 1: Create the 'conversations' table
-- This table will store the list of chat sessions for our sidebar.
CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(), -- A unique ID for each conversation
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE, -- Links to the logged-in user
  title TEXT NOT NULL, -- A title for the conversation (e.g., "First Aid for Burns")
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Step 2: Enable Row Level Security (RLS) on the 'conversations' table
-- This is a critical security measure. It ensures that, by default, no one can see any conversations.
ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;

-- Step 3: Create security policies for the 'conversations' table
-- Policy 1: Allows a user to SELECT (read) ONLY their own conversations.
CREATE POLICY "Allow users to view their own conversations"
ON conversations FOR SELECT
USING (auth.uid() = user_id);

-- Policy 2: Allows a user to INSERT (create) a conversation ONLY if it's for themselves.
CREATE POLICY "Allow users to create their own conversations"
ON conversations FOR INSERT
WITH CHECK (auth.uid() = user_id);

-- Step 4: Create the 'messages' table
-- This table will store every single message from every conversation.
CREATE TABLE messages (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE, -- Links to a specific conversation
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE, -- Redundant user_id for simpler security policies
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant')), -- The sender of the message
  content TEXT NOT NULL, -- The actual text of the message
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Step 5: Enable Row Level Security (RLS) on the 'messages' table
ALTER TABLE messages ENABLE ROW LEVEL SECURITY;

-- Step 6: Create security policies for the 'messages' table
-- Policy 1: Allows a user to SELECT (read) messages ONLY if they belong to that user.
CREATE POLICY "Allow users to view their own messages"
ON messages FOR SELECT
USING (auth.uid() = user_id);

-- Policy 2: Allows a user to INSERT (create) a message ONLY if it's for themselves.
CREATE POLICY "Allow users to create their own messages"
ON messages FOR INSERT
WITH CHECK (auth.uid() = user_id);

```

# Step 3: Set Up the Hospitals Table
This script creates the hospitals table and adds the necessary geospatial index for fast location searching.

```sql
-- This query creates the 'hospitals' table with PostGIS location data.

CREATE OR REPLACE FUNCTION get_hospitals(
  user_lat NUMERIC,
  user_lon NUMERIC,
  search_term TEXT DEFAULT NULL
)
RETURNS TABLE (
  -- The only change is on the line below: INT is now BIGINT
  id BIGINT,
  name TEXT,
  address TEXT,
  phone TEXT,
  distance_meters DOUBLE PRECISION
) AS $$
DECLARE
  query_text TEXT;
BEGIN
  query_text := '
    SELECT
      h.id,
      h.name,
      h.address,
      h.phone,
      ST_Distance(
        h.location,
        ST_SetSRID(ST_MakePoint(' || user_lon || ', ' || user_lat || '), 4326)::geography
      ) AS distance_meters
    FROM
      hospitals AS h';

  IF search_term IS NOT NULL AND search_term != '' THEN
    query_text := query_text || ' WHERE h.name ILIKE ''%' || search_term || '%''';
  END IF;

  query_text := query_text || ' ORDER BY distance_meters';

  IF search_term IS NULL OR search_term = '' THEN
    query_text := query_text || ' LIMIT 3';
  END IF;

  RETURN QUERY EXECUTE query_text;
END;
$$ LANGUAGE plpgsql;
```

# Step 4: Create the Hospital Search Function
This script creates the main PostgreSQL function used to find and search for hospitals.

```sql
-- This script will replace the functions with a version that correctly casts the distance type.

-- First, drop the previous versions to be safe.
DROP FUNCTION IF EXISTS find_nearest_hospitals(double precision, double precision);
DROP FUNCTION IF EXISTS search_hospitals_by_name(text);

-- Re-create the 'find_nearest_hospitals' function with the type cast.
CREATE OR REPLACE FUNCTION find_nearest_hospitals(target_lat float, target_lon float)
RETURNS TABLE (
  id BIGINT,
  name TEXT,
  address TEXT,
  phone TEXT,
  distance_meters NUMERIC
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    h.id,
    h.name,
    h.address,
    h.phone,
    -- THIS IS THE FIX: We cast the result of ST_Distance to NUMERIC
    ST_Distance(
      h.location,
      ST_SetSRID(ST_MakePoint(target_lon, target_lat), 4326)::geography
    )::NUMERIC AS distance_meters
  FROM
    hospitals AS h
  ORDER BY
    h.location <-> ST_SetSRID(ST_MakePoint(target_lon, target_lat), 4326)::geography
  LIMIT 3;
END;
$$;


-- Re-create the 'search_hospitals_by_name' function (no changes needed here, but included for completeness).
CREATE OR REPLACE FUNCTION search_hospitals_by_name(search_term TEXT)
RETURNS TABLE (
  id BIGINT,
  name TEXT,
  address TEXT,
  phone TEXT,
  distance_meters NUMERIC
)
LANGUAGE plpgsql
AS $$
BEGIN
  RETURN QUERY
  SELECT
    h.id,
    h.name,
    h.address,
    h.phone,
    null::NUMERIC AS distance_meters
  FROM
    hospitals AS h
  WHERE
    h.name ILike '%' || search_term || '%';
END;
$$;
```
